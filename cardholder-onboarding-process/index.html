Onboarding Process

General

Any sort of new program onboarding begins by collecting key information about the organization launching the program and the individuals who will have access to the banking as a service platform. This is done to ensure that we provide the right tools for your use case, and also because we provide access to regulated payments and banking services which require us to KYC our customers.

Our sales and account management team should clearly walk you through any information which is necessary to submit for our partners (like your country of incorporation, general program description, and intended account structure) - see Getting Started for more details

Client Entity Onboarding Process

Our technical client onboarding process begins by verifying the identity of the beneficial owner who will be onboarded to the Platform. As QENTA provides access to regulated financial services, QENTA is required by EU payments regulations to verify the identities of all those who access the Platform for anti-money laundering and risk reasons.

The Legal Entity resource is used for verifying the beneficial owner of the client organization. It holds all data necessary to verify a person or organization.

Entity Onboarding Process

Any change to the Legal Entity resource will result in a verification process indicated by the verificationFlow.state attribute switching to PROCESSING. The legal entity can’t be changed unless the process resolves to ERRONEOUS.

Results

Results are sent as LegalEntityEvent (eventType: legal_entity.verification_result) via the webhook interface.

Success

When the verificationFlow.state is SUCCESS then no further changes to the Legal Entity are allowed.

Error

When the verificationFlow.state changes to ERRONEOUS then the verificationFlow attributes erroneous and nextSteps indicate details on the error and how to proceed according to the defined verification flow.

Cardholder Onboarding

A person’s residential address and identity must be verified. When these sub-resources are erroneous then they can be improved / re-verified by using the matching PATCH method.

The following documentation describes the default verification flow for cardholder applicants in the EU.

Residential Address

The residential address country must match one of the payment program’s configured card issuing countries. For example, if your card program is set up to be live in Austria and Germany, we will not be able to issue a card to a cardholder applicant who is located in France.

In case of a mistake in input in the residential address, use this patch method to update the data: Swagger UI

Identity

The name and date of birth of the cardholder applicant will be verified.

If the residential address is verified but the identity is found to have an error, then the cardholder applicant will be prompted to upload an ID document.

An error can be caused by a mistake in the input of a name or date of birth, and also sometimes if there is not sufficient verifiable information about the cardholder applicant. This can especially occur if a person has recently changed addresses.

In case of a typo or mistake in the inputs for identity, use this patch method to update the data: Swagger UI

ID document

Uploading an ID document is only required when the identity input information is found to be erroneous.

The files for the front and back side of a document must be uploaded together.

If documents are re-uploaded, then at least one file must be different to the currently stored ones.

Allowed file types are: JPEG and PNG

Use this patch method to (re)upload the data: Swagger UI

Testing

In the test environment all Legal Entity checks will result in SUCCESS by default. There are keywords for certain fields that will trigger error results:

  • middleName:

    • containsIgnoreCase (“xError:identity”): results in an erroneous identity

    • containsIgnoreCase (“xError:risk”): results in a risk check with total score 99

  • district:

    • containsIgnoreCase (“xError:residentialAddress”): results in an erroneous residential address

  • documentFront:

    • the filename containsIgnoreCase (“xError”): results in an erroneous ID document
