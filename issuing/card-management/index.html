Card Management

General

test change

A Card represents the user’s (cardholder’s) physical or virtual card. Cards spend the funds provided by the linked Balance Account. Multiple Card resources per Balance Account can be created.

When onboarding new clients, the organization must first onboard and upload information about its beneficial owner (as described here) before cardholders, cards and associated balance accounts can be created.

Operations

To see the current version of operations please see Swagger UI

Create Cards

For technical details please see Swagger UI

Cards are issued based on the associated Legal Entity of the cardholder applicant. The residential address country of cardholder applicant the will determine the possible card products which it can obtain. The legal entity’s identity official name will determine the card name on the card.

This can be physical and/or virtual cards.
Virtual cards can be converted to physical cards in a later stage (Convert Cards)

Newly created cards will have status = NEW. They will transition to status = ORDERED automatically. The process triggers two events:

  • card.created (card = NEW)

  • card.updated (card = ORDERED)

If a manual activation flow is configured then the card must be activated (Activate Cards) before it can be used for payments.

Card Delivery Address

Depending on the card program, a cardholder (legal entity) may have a physical and/or virtual card associated with their account. Physical cards require a delivery address associated with the cardholder, while virtual cards do not need a delivery address.

If no address is given for a physical card, then the residential address of the Legal Entity resource is used.

The card delivery is part of the KYC process and must verify the residential address of the individual then the delivery address must be the same as the residential address or null.

Embossed Name

The embossed name on the card will be derived from the cardholder applicant’s submitted name or the name on the ID document. This can be overriden by using the embossedCardHolderName attribute of the card.

If you use multi line embossed names on your cards then '\n' is the line separator.

As the embossed name is a protected detail of the card it won’t be returned by the cards API but is part of the Card Image Visualization API.

Activate Cards

For technical details please see Swagger UI

Available flows are:

  • Active after creation

  • Manual activation with no additional validation parameters

  • Manual activation with CVV validation

  • Manual activation with last four digits validation

Please talk to your implementation manager for configuring the correct card activation flow.
The card activation triggers a card.updated event.

Block / Unblock Cards

For technical details please see Swagger UI and Swagger UI

Cards can be blocked by us or by you. A blocked card can be unblocked.

We block cards according to our Terms and Conditions, e.g. fraud attempts and other suspicious activity.

You can block cards based on your Terms and Conditions and cardholder requests.

A card block triggers a card.updated event.

Cancel Cards

For technical details please see Swagger UI

Cards can be cancelled by us or by you. A cancelled card cannot be activated again.

We cancel cards according to our Terms and Conditions.

You can cancel cards based on your Terms and Conditions and cardholder requests, e.g. the card was lost / stolen.

Lost / Stolen

When the cancel reason is LOST or STOLEN then a new card of the same type (physical / virtual) is automatically created and ordered. If the new card is physical then it will be delivered to the same address as the previous. You will receive the card details by a card created event. The card can be linked to the old one by the previousCardId attribute.

The process will trigger the following events:

  • card.updated (old card, status = CANCELLED)

  • card.created (new card, status = NEW)

  • card.updated (new card, status = ORDERED)

Change the Authentication Details

For technical details please see Swagger UI

Changing the authentication phone number will change the phone number used for the 3DS authentication.

Card features

For technical details please see Swagger UI

The following card features are implemented:

  • ATM enabled: ATM withdrawals can be blocked by setting false

  • NFC enabled: NFC payments can be blocked by setting false

  • E-Commerce enabled: E-Commerce payments can be blocked by setting false

  • Region restriction:

    • No restriction: payments are not blocked due to regional aspects

    • Region: will block any payment with a merchant located outside the defined region (EU + EFTA)

    • Domestic: will block any payment with a merchant located outside the issuing country

Card limits

Limits are configured according to the BIN requirements and your needs. If you need special limits setup by default for your cards please consult your implementation manager.

Limit configurations can be separated using the following criteria:

  • A transaction category

    • ALL - will hit any kind of authorization.

    • CASH - will hit ATM / Cashback authorizations.

    • PURCHASE - will hit other authorizations than CASH (including the purchase part of cashback).

  • A MCC list

    • The limit calculation is restricted to the configured MCC list. If no list is given any MCC will be added to the calculation.

Every limit configuration can configure the following limits:

  • daily limit

  • weekly limit

  • monthly limit

  • yearly limit

  • daily rolling limit

  • weekly rolling limit

  • monthly rolling limit

Normal limits vs rolling limits

Both types of limits can configure:

  • enabled: the limit config can be disabled

  • max spending: sum of all authorized amount < X

  • max transaction count: sum of all authorizations < X

Rolling limits also define the window size. A daily rolling limit with window size four will always look at the last four days (including the present).

Convert Cards

For technical details please see Swagger UI

Virtual cards can be converted to physical. This data can be set using the request object because physical cards have a mandatory delivery address and support the usage of the PIN.

A card conversion will create a new Card resource. Both resources will share the same card number but will have a different CVV and Expiry date if the conversion was later than the same month of the virtual card.

Both will be active and usable by the cardholder with their unique details (CVV / Expiry date). The virtual card will be cancelled when the physical card is activated.

Important:

As both cards share the same card number, some operations will affect both card resources: block/unblock, cancel and change authentication phone number.

The process will trigger the following events:

  • card.created (new card, status = NEW)

  • card.updated (new card, status = ORDERED)

  • card.updated (new card, status = ACTIVE, please see Activate Cards)

  • card.updated (old card, status = CANCELLED)

Card Image Visualization

We do not host the image of a card ourselves. Please contact your implementation manger on the image file to use and host it on your own.

To show the image with the card data we provide an endpoint to fetch it. The fetch operation must be done from the cardholder’s device. The data must not be routed through your system if you are not PCI DSS certified.

The fetch is unauthenticated and therefore must be initiated through the trusted environment.

Our recommended flow is:

  • The cardholder starts the show card process

  • You authenticate the cardholder (e.g. fingerprint)

  • You initiate the call using your backend

  • You fetch the data on the cardholders device and combine it with the card image

Card visualizations can’t be created for physical cards when they are configured with a manual activation flow using a card detail (like the last four digits) and are not activated yet.

Initiate a Visualization

For technical details please see Swagger UI

This is an authenticated backend to backend call. It creates an OTC (One Time Code) which is valid for 15 seconds.

Fetch the Card Data / Show the Image

For technical details please see Swagger UI

This is an unauthenticated call that must be done from the cardholder’s device. Use the same deviceId as for the initiate call and the provided OTC (One Time Code) of the initiate call response.

Combine the card data with the card image and show it to the cardholder.

A fetchCardData call is only valid when

  • the OTC (One Time Code) and deviceID match.

  • the call is done within 15 sec of the initiate call.

  • the call is done the first time.

Card Name

If you use multi line card names on your cards then '\n' is the line separator.

Investigate Problems/Results

For technical details please see Swagger UI

Use the GET API to keep track of what happened to the initiated visualization. It provides information if it was fetched successfully or not and how often it was fetched.
