Events

Events

API Versioning

Changes to the API result in a new version. The semantic versioning scheme is applied.

MAJOR.MINOR

MAJOR is increased when incompatible changes of the API are made public

MINOR is increased when the API is extended by new features without breaking the existing API or API compatible bug fixes are applied.

When we change APIs, we try to keep all consumers running. Our goal: API changes SHOULD NOT break backward compatibility.

Event messages can never be changed once created. This means that subsequent updates to the API (new version) do not alter existing event messages. Fetching older events using a newer API version also has no impact on the structure of the events. Hence older events can be read with a newer API version.

Delivery and Retries

The Qenta Issuing Platform attempts to deliver your events for up to three days with an exponential back off.

You can use the Event resource to query for missed events or to reconcile the data over a certain time period.

Events are not rescheduled/retried if no Webhook resource is configured for the event type and is active during creation time.

We treat the following as a successful delivery:

  • your webhook endpoint responds with a 2xx HTTP status code

  • within 3 seconds

Handle Duplicates

It is possible that you receive the same event more than once. Make sure your event processing logic can process duplicate events.

Order of Events

We do not guarantee delivery of events in the order in which they are generated. As long as you don’t miss events you will receive events in order.

If your webhook endpoint does not meet the requirements for a successful delivery we postpone delivery by scheduling a retry with an exponential back off time (max back off time = 60min). Notice, we DO NOT stop sending events in the meantime.

Security

For security reasons, you have to provide a HTTPS endpoint for your webhook. We validate that the connection to your server is correctly set up with a valid server certificate. We require at least TLS v1.2 for encryption.

Additionally only trust events coming from our IP addresses.

Signature

For each request, we create a signature based on the shared secret of your webhook. The signature is found in the HTTP header with key X-QENTA-SIG. The value has the following format: xq:timestamp:hash.

For Example: X-QENTA-SIG=xq:1632827162:17325e0ae016d1af8006bc6c2c86394c468e39031762a3c221f90e4b1ddf3dfa

Validate the content by recalculating the hash value and comparing it to the one of the header.

In order to calculate the hash you need to take the request body, append the timestamp from the signature as it is and hash this string with the HmacSHA256 algorithm. The resulting hash is taken to lower case for comparison.

Pseudo-code to generate the hash
String timestamp = "1632827162"
String beforeHash = requestBody + timestamp;
String hash = hmacSha256Digest(beforeHash, sharedSecret);

Event Object

Qenta sends you an object having a list of events. The basic object structure of an event is as follows:

{
  "events": [
    {
      "id": "${event_id}",
      "eventType": "${event_type}",
      "apiVersion": "1.0",
      "paymentProgramId": "${payment_program_id},
      "createdAt": "2021-09-28T11:06:02Z",
      "sendingAttemptCount": 2,
      "lastSendingAttemptAt": "2021-09-28T11:10:02Z",
      "data": {
      }
    }
  ]
}

The data field holds the details about the event.

On the first delivery attempt, sendingAttemptCount will be set to zero and lastSendingAttemptAt will be null.

Types of Events

For payloads, please refer to https://dev.qenta.com/issuingapidoc/

  • transaction.authorization (data holds a Transaction object)

  • transaction.authorization_reversal (data holds a Transaction object)

  • transaction.capture (data holds a Transaction object)

  • transaction.capture_reversal (data holds a Transaction object)

  • account.adjustment (data holds an AccountAdjustment object)

  • legal_entity.verification_result (data holds a LegalEntityVerificationResult object)
